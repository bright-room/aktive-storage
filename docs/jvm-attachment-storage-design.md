# 構想ドキュメント: JVM/Kotlin 向けファイル添付基盤（ActiveStorage 相当）

## 1. 概要

Ruby on Rails の ActiveStorage に相当する「モデルにファイルを紐付け、クラウド／ローカルに保存し、署名付き URL で配信する」仕組みを、特定の Web フレームワークに依存しない純粋な JVM（Java / Kotlin）ライブラリとして構築する構想。素の Java/Kotlin アプリから使えるコアを土台に、Ktor Plugin や Spring Boot Starter を上に重ねていく。汎用ライブラリとして Maven Central で配布する。

本ドキュメントは設計の方針と全体像を示すもので、具体的な実装コードは含めない。


## 2. 背景と動機

### 課題

JVM エコシステムには、ActiveStorage のように「フレームワーク本体に統合され、ワンライナーでファイル添付が完結する」仕組みが存在しない。現状の選択肢は次のいずれかに偏っている。

- ストレージへの保存処理だけを提供する低レイヤーな部品（各クラウド SDK、jClouds 等）。モデルとの紐付けやライフサイクル管理は自前。
- Spring に密結合した既製品（Spring Content）。Ktor や素の Kotlin では使えない。

そのため Ktor 環境では、毎回プロジェクトごとに「アップロード受け取り → ストレージ保存 → メタ情報の DB 管理 → 配信 URL 発行」のボイラープレートを書き起こすことになる。

### 方針

ActiveStorage の本質的な構造は「**ストレージ抽象層**」と「**メタデータ層**」の分離にある。この 2 層構造を踏襲し、既製の SDK で賄える「実装」は活用しつつ、フレームワーク非依存ゆえに既製品が無い「抽象（ポート）」と「紐付け」だけを自作する。


## 3. 確定した設計判断

検討の結果、次の方針を確定とする。以降の章はこの判断に沿って記述する。

- **配布形態**: 汎用ライブラリとして Maven Central に公開する（リリース実績あり）。各機能は独立した artifact に分割し、利用者は必要なものだけ依存に追加する。
- **命名・座標**: ライブラリ名は `aktive-storage`。groupId にプロジェクト名まで含めて `net.bright-room.aktive-storage` とし、artifactId は概念ごとに短く保つ（`core`、`metadata-exposed`、`storage-s3` 等）。座標は `net.bright-room.aktive-storage:<artifact>:<version>`。
- **バージョニング**: Maven Central 推奨に従いセマンティックバージョニング（SemVer）。全モジュールを同一バージョンでロックステップ運用し、`net.bright-room.aktive-storage:bom` を提供してバージョン整合を取れるようにする。
- **モジュール構成**: Gradle のマルチプロジェクトで、`core` を中心に各実装・各連携をアドオン的なモジュールとして積む（詳細は 5 章）。
- **ストレージ抽象**: jClouds は採用しない。自前の抽象ポートを定義し、各クラウドのネイティブ SDK の薄いアダプタで実装する。マルチクラウド対応は自前ポート経由で実現する。
- **実装言語の優先**: コアは Kotlin・coroutine ベース。S3 アダプタは Kotlin ネイティブの AWS SDK for Kotlin（suspend 関数）を優先し、コアの非同期モデルと噛み合わせる。Kotlin SDK が無いクラウドは Java SDK をラップし、ブロッキングを `Dispatchers.IO` 等へ逃がす。
- **ORM 依存**: 抽象化する。ただし ORM 全体ではなく、ライブラリが必要とする最小操作だけに絞ったポートを切り、各 ORM 用アダプタを別モジュールで提供する。
- **フレームワーク対応の順序**: まず Pure Java / Pure Kotlin で使えるコアを完成させ、その後 Ktor Plugin、Spring Boot Starter を追加モジュールとして提供する。


## 4. 設計原則

- **フレームワーク非依存**: コアは Web フレームワーク・DI コンテナに依存しない。フレームワーク連携は薄いアダプタ層として分離する。
- **ストレージ中立**: 保存先（ローカル / S3 / GCS / Azure / S3 互換）は設定で差し替え可能にし、アプリのコードからは隠蔽する。
- **抽象は自前、実装は委譲**: 抽象（ポート）は自分で定義して管理下に置き、ワイヤープロトコル（署名計算・マルチパート等）は既存 SDK に任せてポートの裏に隠す。
- **キーは確定して永続化、再導出しない**: ストレージ上のオブジェクトキーは生成時に確定して metadata に永続化し、読み出し時に業務状態（ユーザー ID や元ファイル名など）から二度と計算しない。これが守るべき不変条件。既定のキーは不透明・推測不能なランダムトークンとし、名前衝突と列挙を防ぐ。元ファイル名はメタデータとして退避する。ただしキー生成は差し替え可能とし（プレフィックス付与や content-hash 等）、業務的な意味を持たせたい場合も「生成時に決めて固定する」ルールの範囲で許容する。
- **アプリサーバーを経由しない大容量転送**: アップロード・ダウンロードの実体転送は、可能な限りクライアントとストレージの間で直接行わせ、アプリは署名付き URL の発行と検証に専念する。
- **遅延と再利用**: 画像バリアントは要求時に生成し、生成結果は決定的なキーで保存して以降は使い回す。
- **段階的に薄く**: 最初から全機能を作らず、MVP から積み上げる。


## 5. モジュール構成

Gradle マルチプロジェクトとして構成し、各モジュールを独立した artifact として publish する。利用者は必要なものだけを依存に足す（kotlinx 系や Micrometer、Ktor 本体と同じ配布スタイル）。

| モジュール | 役割 | 依存の方向 |
| --- | --- | --- |
| `core` | ポート定義（`StorageService`、メタデータ SPI、バリアント API）と高レベル添付 API。フレームワーク・ORM・クラウド SDK に一切依存しない | 何にも依存しない（中心） |
| `storage-s3` | S3／S3 互換ストレージのアダプタ（AWS SDK for Kotlin を利用） | `core` |
| `storage-fs` | ローカルファイルシステムのアダプタ | `core` |
| `storage-gcs` / `storage-azure` | 各クラウドのアダプタ（Java SDK をラップ） | `core` |
| `metadata-exposed` | Exposed 用メタデータ永続化アダプタ | `core` |
| `metadata-jooq` / `metadata-jpa` | 各 ORM 用アダプタ | `core` |
| `variant-thumbnailator` / `variant-scrimage` | 画像変換アダプタ | `core` |
| `integration-ktor` | Ktor Plugin。マルチパート受信・ルーティングとの接続 | `core`（+ 利用者が選んだアダプタ） |
| `integration-spring-boot-starter` | Spring Boot Starter。自動構成 | `core`（+ アダプタ） |

依存は常に各アダプタ → `core` の一方向。`core` がアダプタを知ることはない（依存性逆転）。これにより、ストレージ・ORM・フレームワークいずれも利用者側の選択で差し替え可能になり、「Pure からはじめて連携モジュールを後から足す」段階計画がモジュール追加だけで成立する。

各モジュールの Maven 座標は `net.bright-room.aktive-storage:<artifactId>`（例: `net.bright-room.aktive-storage:core`、`net.bright-room.aktive-storage:storage-s3`）。全モジュールを同一バージョンでリリースし、`net.bright-room.aktive-storage:bom` で版を揃える。


## 6. アーキテクチャ全体像

レイヤー構成（上位がアプリ寄り、下位がインフラ寄り）と、それぞれの調達方針。

| レイヤー | 役割 | 調達方針 |
| --- | --- | --- |
| アプリ連携アダプタ | Ktor / Spring 等との接続 | 自作（`integration-*`） |
| 添付 API（コア） | `attach` / `urlFor` / `detach` 等の高レベル操作 | 自作（`core`） |
| メタデータ層 | Blob・Attachment の永続化とライフサイクル | ポートは自作、実装は ORM アダプタ |
| バリアント層 | 画像変換と変換結果のキャッシュ | ポートは自作、変換は画像ライブラリ |
| ストレージ抽象層（Service） | 保存・取得・削除・署名 URL 発行の共通 IF | ポートは自作 |
| ストレージ実装 | ローカル FS / S3 / GCS / Azure | 各クラウドのネイティブ SDK |

ポイントは、横断的な抽象（ポート）はすべて自前で持ち、各クラウド・各 ORM の固有処理だけを既存 SDK に委譲する点。出来合いのマルチクラウド抽象（jClouds）は、自前ポートと二重になるため採用しない。


## 7. コンポーネント設計

### 7.1 ストレージ抽象層（Service）

保存先の違いを吸収する共通インターフェース（ポート）を自前で定義する。想定する操作は次の通り。

- バイト列の保存（キー指定、メタ情報付き）
- バイト列の取得（ストリーム）
- 存在確認・削除
- 署名付き URL の発行（アップロード用 PUT / ダウンロード用 GET、有効期限付き）

各実装はネイティブ SDK の薄いアダプタとする。S3／S3 互換は Kotlin ネイティブの AWS SDK for Kotlin を用い、suspend 関数をそのままコアの coroutine モデルに繋ぐ。Kotlin SDK が無いクラウドは Java SDK をラップし、ブロッキング呼び出しを `Dispatchers.IO` 等へ退避させる。署名 URL の発行も各 SDK の機能に委譲する。

「複数の保存先へ同時書き込みする実装（移行用ミラー）」も、将来的にこのポートの一実装として追加できる余地を残す。

このポートは **キーを生成も導出もしない**。常に「すでに確定したキー」を受け取って保存・取得するだけの素朴な口に保つ。キーを決める責務（思想）は次のメタデータ層に集約し、ストレージ抽象は dumb なまま保つ。

### 7.2 メタデータ層

ストレージ上の実体と、アプリのドメインレコードを結びつける情報を永続化する。ActiveStorage の二段構成を踏襲する。

- **Blob**: ファイル実体のメタ情報（ストレージキー、元ファイル名、コンテンツタイプ、バイトサイズ、チェックサム、保存先サービス名）。実体バイトは保持しない。
- **Attachment**: Blob とドメインレコードを結ぶ関連。どのレコードの何という名前の添付か（例: User の `avatar`）を表す。1 つの Blob を複数レコードから参照できる構造も考慮する。

**キー生成はこの層のストラテジ**として持つ。`attach` 時に一度だけキーを確定させ、Blob に永続化する。既定の戦略は不透明・推測不能なランダムトークン。差し替えで「プレフィックス＋ランダムサフィックス」（例: `avatars/<ランダム>` ＝ ライフサイクルやアクセスポリシーをプレフィックス単位で当てつつ、衝突回避と推測不能性は維持）や content-hash なども選べるようにする。完全な意味付きパスは外部システムとの厳密な互換などに限り、いずれの戦略でも「生成時に決めて固定し、業務状態から再導出しない」不変条件は共通とする。

ORM 依存は抽象化するが、**ORM 全体を抽象化しない**のが原則。ライブラリが本当に必要とする操作（Blob/Attachment の save・find・delete 程度）だけに絞った狭いポートを `core` に置き、`metadata-exposed` などのアダプタで実装する。抽象を広げすぎると重くなり、ORM の利点も殺すため、必要最小限にとどめる。

**トランザクション境界**は唯一の難所として明示的に割り切る。添付の書き込みはアプリの業務トランザクションに相乗りさせたい（添付だけ commit され本体が rollback、を防ぐ）ことが多いが、トランザクション管理の作法は ORM ごとに異なる。そこで、`core` はトランザクションを知らない素朴な設計とし、相乗りが必要な場合は各 integration モジュール（Ktor/Spring）側で境界を合わせる方針とする。透過的な統合を core で目指すと抽象が一気に重くなるため、その複雑さは連携層に押し出す。

### 7.3 URL 発行・配信戦略

ActiveStorage の 2 モードを踏襲する。

- **リダイレクトモード（既定）**: アプリは署名付きの「リダイレクト用 URL」を返す。クライアントがアクセスすると、アプリは署名を検証し、ストレージの期限付き実 URL へ 302 で転送する。実体転送はクライアントとストレージの間で直接行われ、アプリに負荷がかからない。
- **プロキシモード**: アプリが実体をストリーム配信する。CDN の前段に置く場合や、配信時に独自の制御を挟みたい場合に使う。

署名 URL に埋め込む参照（Blob の識別子）は改竄・推測を防ぐため署名し、有効期限を持たせる。

### 7.4 バリアント（画像変換）層

リサイズ・フォーマット変換などの派生画像を扱う。

- 変換指定の内容からダイジェスト（ハッシュ）を計算し、それを派生画像の決定的なストレージキーとする。
- 要求時にキーの存在を確認し、無ければ元画像を取得して変換・保存（遅延生成）。以降は同じキーを使い回す。
- 配信はオリジナルと同じ URL 発行・配信戦略に乗せる。
- 変換エンジンは Thumbnailator や scrimage 等の JVM 画像ライブラリを `variant-*` アダプタとして差し込む。重い変換をアプリから切り離したい場合は、外部の画像変換サービス（imgproxy 等）へ署名付き URL で委譲する構成も選択肢とする。

### 7.5 ダイレクトアップロード

大きいファイルをアプリサーバーを経由させずに保存する経路。

1. クライアントがファイルのメタ情報（名前・サイズ・チェックサム・タイプ）をアプリに送る。
2. アプリは Blob レコードを先に作成し、署名付きの PUT URL と Blob の署名済み識別子を返す。
3. クライアントはストレージへ直接 PUT する。
4. フォーム送信時に署名済み識別子を渡し、アプリが Attachment を作成して紐付ける。

この経路は MVP では必須ではないため、段階的に追加する。


## 8. データモデル（概念レベル）

実カラム定義ではなく、保持すべき情報の概念整理。

**Blob 相当**

- ストレージキー（ランダムトークン、配信パスの実体）
- 元ファイル名
- コンテンツタイプ
- バイトサイズ
- チェックサム（整合性検証用）
- 保存先サービス名
- 作成日時

**Attachment 相当**

- 添付名（例: `avatar`, `documents`）
- 紐付け先レコードの種別と識別子
- 参照する Blob
- 作成日時

**Variant 記録（任意）**

- 元 Blob への参照
- 変換指定のダイジェスト

Variant の記録テーブルは、生成済みバリアントの追跡を厳密に行いたい場合のオプション。MVP では「キーの存在確認」だけで代替できる。


## 9. 主要フロー

### 9.1 アップロード（サーバー経由・基本形）

マルチパートで受信 → チェックサムとサイズを算出 → ランダムキーを採番 → Service 経由でストレージへ保存 → Blob 作成 → 対象レコードに対する Attachment 作成。

### 9.2 アップロード（ダイレクト）

7.5 の手順に従い、実体転送をクライアント↔ストレージ間に逃がす。

### 9.3 取得（リダイレクトモード）

アプリが署名付きリダイレクト URL を返す → クライアントがアクセス → 署名検証 → Service が期限付き実 URL を生成 → 302 でクライアントを転送 → クライアントがストレージから直接ダウンロード。

### 9.4 削除

Attachment の削除と、参照されなくなった Blob・ストレージ実体の削除を分けて扱う。実体削除は他からの参照が無いことを確認したうえで、非同期ジョブに逃がすことを検討する（削除漏れと即時削除の負荷を避けるため）。


## 10. 技術選定の指針

| 対象 | 採用 | 備考 |
| --- | --- | --- |
| ストレージ抽象 | 自前ポート | jClouds は不採用（自前抽象と二重になるため） |
| S3 アダプタ | AWS SDK for Kotlin | coroutine／suspend ベースでコアと整合 |
| その他クラウドアダプタ | 各クラウド Java SDK | ブロッキングは `Dispatchers.IO` へ退避 |
| 署名付き URL | 各 SDK の署名機能 | ポートの裏に隠蔽 |
| 永続化 | ORM ポート + アダプタ | Exposed を第一候補、jOOQ/JPA も追加可 |
| 画像変換 | Thumbnailator / scrimage | 外部サービス（imgproxy 等）委譲も選択肢 |
| Web 連携 | Ktor Plugin / Spring Boot Starter | Pure コアの上に後から追加 |


## 11. スコープと段階的実装計画

モジュール追加として段階を積む。

### フェーズ 1（MVP）

- `core`（ポート + 添付 API）
- `storage-fs` と `storage-s3`（S3 互換含む）
- `metadata-exposed`
- サーバー経由アップロード
- リダイレクトモードでの取得（署名・有効期限つき）

これだけで「モデルにファイルを紐付けて保存し、安全な URL で配信する」という中核体験が Pure Kotlin で成立する。

### フェーズ 2

- `integration-ktor`（Ktor Plugin）
- `variant-*`（画像バリアント：遅延生成 + キャッシュ）
- プロキシモード配信
- 削除ライフサイクルの整理（孤立 Blob の回収）

### フェーズ 3

- `integration-spring-boot-starter`
- ダイレクトアップロード
- 追加の `storage-*`（GCS / Azure）と `metadata-*`（jOOQ / JPA）
- 保存先のミラーリング（移行支援）
- メタデータ解析（画像の寸法抽出など）


## 12. 非機能・考慮事項

- **セキュリティ**: 署名 URL の有効期限管理、署名鍵のローテーション、コンテンツタイプの検証、アップロードサイズ上限、実行可能ファイル等の扱い。
- **整合性**: チェックサムによる破損検知。DB とストレージの不整合（片方だけ成功）への対処方針（先に Blob 行を作る順序、リトライ、孤立データの回収ジョブ）。
- **トランザクション**: 7.2 の通り、core はトランザクション非依存とし、業務トランザクションへの相乗りは integration 層で実現する。アダプタ間でこの責務分担を一貫させる。
- **削除と GC**: 参照されなくなった実体の回収戦略。即時削除と遅延削除のトレードオフ。
- **並行性**: 同一バリアントの同時生成（重複生成の許容か、ロックか）。coroutine ベースのため、ブロッキング SDK 呼び出しのディスパッチャ分離も設計に含める。
- **可搬性**: ストレージ移行時のキー設計の安定性。保存先サービス名を Blob に持たせることで、混在・移行を許容する。
- **観測性**: アップロード／配信の成否、変換時間、ストレージ使用量のメトリクス。


## 13. 将来拡張の余地

- ウイルススキャン連携（保存後の非同期チェック）。
- 動画・音声向けのメタデータ抽出やサムネイル生成。
- アクセス制御の高度化（レコードの可視性とリンクした署名）。
- 全文検索・インデックス連携（ドキュメント系）。
- Kotlin/Native 等、追加プラットフォーム対応（コアのフレームワーク非依存性を活かす）。


## 14. 残された論点

主要な方針は 3 章で確定済み。引き続き検討が必要なのは以下。

- メタデータ用ポートの粒度。最小操作に絞る方針は確定しているが、「最小」の具体的な境界（バッチ削除や一覧取得を含めるか等）は実装しながら詰める。当たりとして、ActiveStorage が実際に提供している「レコード単位の一覧」「purge / purge_later（一括削除）」「unattached（孤立 Blob の発見）」を初期スコープの目安とする。なお ActiveStorage ではこれらが Active Record モデルゆえ自動で備わるのに対し、本ライブラリは ORM を抽象化するぶん、何を通すかを明示的に選ぶ必要がある（これはポータビリティと引き換えの本質的なトレードオフ）。
- トランザクション相乗りを各 integration でどこまで自動化するか（明示 API にするか、フレームワークの仕組みに乗せて透過にするか）。
- バリアントの追跡を専用テーブルで厳密に行うか、ストレージのキー存在確認で済ませるか。
